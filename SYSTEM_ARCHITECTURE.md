# agentic-sdlc — System Architecture

## Overview

agentic-sdlc is built on three layers: a human interface (Slack), an agent runtime (Claude API + GitHub API), and a state store (GitHub). These three layers are deliberately separated — the runtime is framework-agnostic and could be exposed through a web portal or GitHub Actions just as easily as Slack.

```
Human Interface Layer       Agent Runtime Layer         State Layer
────────────────────        ───────────────────         ───────────
Slack (primary)             Claude API                  GitHub
  #all-health360              claude-sonnet-4-6           specs/ (markdown)
  #feature-*                  claude-haiku-4-5            work-items/ (issues)
  Button interactions         (intent classification)     branches (phase state)
                                                          PRs (approval gates)
GitHub Actions (future)     GitHub API (Octokit)
Web portal (future)           Read specs
                              Save drafts
                              Open review requests
                              List in-progress features
```

## The agent workflow

Every feature follows the same sequence. Each phase is a Slack channel conversation between a human and an AI specialist. No phase starts until the previous one is approved.

```
ENTRY POINT
  #all-health360 → Concierge agent
  Anyone asks "what can I do?" → role-aware response + current feature state

PHASE 1: Product Spec
  Channel: #feature-<name>
  Human: Product Manager
  Agent: pm agent (claude-sonnet-4-6)
  Output: <feature>.product.md
  Gate: PM explicit approval → review request opened

PHASE 2: Design Spec  [planned]
  Channel: #feature-<name>
  Human: UX Designer
  Agent: design agent
  Output: <feature>.design.md
  Gate: Designer explicit approval → review request opened

PHASE 3: Engineering Spec  [planned]
  Channel: #feature-<name>
  Human: Software Architect
  Agent: architect agent
  Output: <feature>.engineering.md
  Gate: Architect explicit approval → review request opened

PHASE 4: Build  [planned]
  Work items generated by pgm agent from engineering spec
  Domain agents (backend, frontend) claim and execute work items
  Gate: QA agent sign-off

PHASE 5: QA  [planned]
  QA agent tests against original product spec
  Gate: All acceptance criteria met
```

## Phase state detection

The system determines what phase a feature is in by reading GitHub — no separate database needed.

| GitHub state | Feature phase |
|---|---|
| Branch `spec/<feature>-product` exists, file not on main | Product spec in progress |
| `<feature>.product.md` merged to main | Product spec approved, awaiting design |
| Branch `spec/<feature>-design` exists, file not on main | Design in progress |
| `<feature>.design.md` merged to main | Design approved, awaiting engineering |
| `<feature>.engineering.md` merged to main | Engineering spec approved, build phase |

## Key components

### Concierge agent (`agents/concierge.ts`)
The front door. Lives in the main Slack channel. Reads current feature state from GitHub on every message. Responds in plain English based on the person's role. Knows the full agent roster and what phase each role belongs to. Never uses technical jargon (no "PR", "branch", "commit").

### pm agent (`agents/pm.ts`)
Shapes feature briefs into product specs through conversation. Asks clarifying questions, pushes back on conflicts with product vision or architecture, surfaces edge cases. Auto-saves a draft to the feature branch after every substantive response. Opens a review request only on explicit approval signal. Reads the current saved draft at the start of every message so it always continues from the latest state — conversation history is supplementary, the draft is the source of truth.

### Intent classifier (`runtime/agent-router.ts`)
Uses `claude-haiku-4-5` (faster, cheaper) to classify which agent should handle a message and what phase the feature is in. Routes to the right agent. On first message in a thread, surfaces a confirmation UI with Slack buttons. Confirmed agent is persisted to disk so bot restarts don't re-ask.

### Context loader (`runtime/context-loader.ts`)
Loads agent context from GitHub on every message: product vision, feature conventions, system architecture, and the current draft spec for the feature. All context is fresh on every call — no caching.

### GitHub client (`runtime/github-client.ts`)
All GitHub operations: read files (any branch), save draft specs (commit to feature branch without opening review request), open review requests, list in-progress features from branch names.

### Conversation store (`runtime/conversation-store.ts`)
In-memory conversation history per Slack thread (keyed by `thread_ts`). Confirmed agent per thread, persisted to `.confirmed-agents.json` on disk so restarts don't wipe it. Production target: Redis or Neon for full persistence.

## Design principles

**GitHub is the source of truth.** All spec state lives in GitHub. The Slack bot is stateless beyond conversation history. Any agent can be re-run against the same GitHub state and produce consistent results.

**Agents are stateless functions.** Each agent reads from GitHub, produces a response, optionally writes back to GitHub, and exits. No agent holds state between calls beyond what's in the conversation store.

**Slack for humans, GitHub for machines.** Humans interact through Slack in plain English. Agents read from and write to GitHub. The two layers are connected only at explicit handoff points (review requests, approval signals).

**Human approval at every gate.** No phase advances automatically. An AI agent can produce a draft, surface it, and ask for approval — but the human always makes the decision to advance.

**Framework-agnostic runtime.** The agent logic (`agents/`, `runtime/`) has no dependency on Slack. `interfaces/slack/` is one delivery mechanism. A GitHub Actions trigger or a web portal could invoke the same runtime without touching the agent logic.

## Technology choices

| Component | Technology | Why |
|---|---|---|
| Primary AI | claude-sonnet-4-6 | Best reasoning for spec shaping and architectural decisions |
| Intent classification | claude-haiku-4-5 | Faster and cheaper for classification-only tasks |
| Human interface | Slack (Socket Mode) | Where teams already work; no public URL needed for Socket Mode |
| GitHub integration | Octokit REST | Official GitHub SDK; full API coverage |
| State persistence | GitHub branches + files | Version-controlled, auditable, no extra database |
| Conversation history | In-memory + disk (dev) / Redis (prod) | Fast for dev; Redis for production reliability |

## What's not in this repo

- The apps being built (those live in product repos like `agentic-health360`)
- CI/CD pipelines for those apps (that's `agentic-cicd`)
- Design tokens and brand assets (that's `agentic-brand`)
- The product roadmap for any specific app (that lives in each product repo's `specs/`)
