The thesis

An AI specialist at every phase.
Deterministic by construction.

Every phase of the SDLC has a dedicated AI agent with deep domain expertise. Every audit that gates a decision is a pure function — same input, same output, always. Every behavior that must reliably occur is enforced by the platform, not instructed in a prompt.

The result: a software pipeline that humans steer but never have to babysit.

The agent loop

Every LLM call = persona + fresh context + history.

Agents are stateless. Every turn, the platform rebuilds the prompt from scratch — current persona, freshly-loaded specs from GitHub, conversation history. No hidden state in the model. No drift between turns.

1
Persona

System prompt for the resolved agent. Domain expertise, output contract, domain boundary ("what you never own"). Built fresh per call.

2
Fresh context

Current product / design / engineering spec, brand tokens, feature phase — all read from GitHub at call time. Never cached.

3
History

Slack thread history for the feature channel. The only mutable input. Everything else is canonical state on disk.

Single routing authority

resolveAgent() is the only source of truth.

The rule

Phase implies agent. Always.

Every feature channel routing decision flows through resolveAgent(featureName). It reads the feature's phase from GitHub — based on which spec branches exist — and deterministically maps it to the canonical agent.

If persisted state disagrees with GitHub, GitHub wins. Stale routing is structurally impossible.

Deterministic audits

Same input. Same output. Always.

Every audit that gates a decision is a pure function — parsing, counting, string matching, diff comparison. No LLM in the gate path. Ask an agent the same question 10 times — the findings are identical.

▯
auditPmSpec

Parses the product spec. Flags vague timing, vague language, missing acceptance criteria, structural gaps.

▯
auditDesignSpec

Validates UX flows against the product spec. Audits brand token usage against BRAND.md. Detects open questions.

▯
auditEngineeringSpec

Checks engineering spec against approved design + product. Flags conflicts, gaps, unresolved decisions.

LLM-based rubrics may exist as a secondary enrichment layer — but never as the primary gate. Suggestion engines don't gate decisions.

Platform enforcement

Behaviors that must occur are enforced — not instructed.

Prompt rules are probabilistic. Anything trust-critical is detected by the platform after the agent runs. If the required behavior didn't happen, the platform corrects it before the response leaves the system.

Hedge gate

Opinionated recommendations only

Counts required-format markers in agent output. If the agent hedged ("clarify whether...", "consider X or Y"), the platform re-runs with explicit instruction to recommend.

enforceNoHedging() · countRecommendations()
Finalization gate

No approval over open questions

Hard-blocks finalize_*_spec tool calls if blocking questions remain. Runs upstream-spec auditors retroactively — approved specs are not exempt.

extractBlockingQuestions() + auditUpstreamSpecs()
Escalation gate

Cross-domain gaps go to the right owner

If the design agent finds a product-scope gap, the platform auto-triggers escalation back to PM. The agent doesn't have to remember to call the tool.

auto-trigger via [type: product] detection
Cross-surface SSOT

Same state, same answer, everywhere

State queries and finalization gates derive from the same canonical function — buildReadinessReport(). "Nothing blocking" and "approved" never disagree.

runtime/readiness-builder.ts
Spec write ownership

Every agent owns one spec. Resolved decisions land only there.

AgentOwned specPath on disk
PMProduct specfeatures/<f>/<f>.product.md
DesignerDesign specfeatures/<f>/<f>.design.md
ArchitectEngineering specfeatures/<f>/<f>.engineering.md

Cross-agent writes are allowed only for flagged-open-item handoffs (TODO sections, "design assumptions to validate"). Resolved content from agent X never lands in agent Y's spec. Enforced at PR time by tests/invariants/spec-write-ownership.test.ts.

Cross-agent parity

Every enforcement mechanism exists in every analogous path.

When a gate is added for one agent, it is added for every analogous agent in the same commit. Mechanisms never live in just one path — the platform structurally maintains parity across the roster.

Examples

What "parity" means

Hedge gate exists in PM, Designer, Architect, Tech Lead — not just one. Escalation writeback exists in every escalation path (Designer→PM, Architect→PM, Architect→Designer, QA→Architect).

Closed loop

Multi-tenant by config

A new team onboards in under an hour. Zero code changes.

Every product-specific coordinate — name, GitHub repo, Slack channels, spec paths, brand file location — lives in WorkspaceConfig. The platform reads from config; no string literal in the codebase refers to a specific product.

.env per tenant
WORKSPACE_NAME=health360 GITHUB_OWNER=acme GITHUB_REPO=health360-product SLACK_MAIN_CHANNEL=all-health360 SLACK_FEATURE_CHANNEL_PREFIX=feature- BRAND_FILE_PATH=BRAND.md SPEC_DIR=features/

Onboarding a second team requires editing one .env file. If it required a TypeScript change, that's a bug.

Domain boundaries

What every agent never owns.

PM

Never: UI copy, visual design, technical implementation, work distribution.

Designer

Never: Product behavior decisions, technical architecture, work distribution.

Architect

Never: Product or design decisions, individual code commits, deployment.

Tech Lead

Never: Architecture decisions, code authorship, QA verdicts.

Coder

Never: Spec changes, work item creation, deployment decisions.

QA

Never: Spec authorship, code authorship, deployment authority.

Every agent's system prompt includes a ## Domain boundary — what you never own section. Crossing it triggers structural escalation.
