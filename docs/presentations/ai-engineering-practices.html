Artifact 1 of 3

CLAUDE.md — the onboarding contract

Challenges it solves
1 Claude opens every session with zero knowledge of your project — conventions, constraints, architecture, what not to do.
2 It makes fast, reasonable-looking decisions that contradict choices from last sprint. No one told it they existed.
3 Every engineer gets a different version of Claude depending on what they happen to say first.
What we put in place
Read automatically at the start of every session, before any action
// CLAUDE.md — checked into repo root
§1 What is this repo? — what it is + what it is NOT
§2 Core Principles — each rule + violation examples inline
§3 Before Writing Any Code — 4-question checklist
§4 Definition of Done — change type → required doc update
§5 Testing Discipline — second-piece rule + mocking pattern
§6 Behavioral Contracts — autonomous, durable, self-correcting
§7 Subagent Strategy — delegate vs. keep in main context
First line: "Read this before doing anything. These are non-negotiable constraints."
CLAUDE.md is not a README. It's not documentation for humans. It's an instruction set for Claude — loaded before every session starts.
Artifact 2 of 3

Memory system — persistent learning across sessions

Challenges it solves
1 Corrections made in one session vanish. Claude repeats the same mistake next week — you have to re-explain everything.
2 CLAUDE.md holds project rules, but can't hold every user preference, confirmed approach, or evolving project state.
3 Without a record of what worked, Claude drifts — it avoids past mistakes but also drifts away from validated approaches.
What we put in place
Read when relevant context is needed; written after every correction or validation
① MEMORY.md — index file, loaded every session. Links to topic files. Max 200 lines.
② Per-topic files — each memory in its own file with type, description, and why + how to apply.
feedback How to approach work — corrections and confirmed approaches. Includes why so edge cases can be judged.
user Role, goals, knowledge level. Tailors explanations to what the user already knows.
project Ongoing decisions, constraints, deadlines. What's driving the work right now.
reference Where things live — Linear project, Grafana board, Slack channel to watch.
Saved after every correction. The key discipline: save both corrections AND confirmations. Saving only mistakes makes Claude overly cautious and it drifts away from validated approaches.
Artifact 3 of 3

CI enforcement — the contract with teeth

Challenges it solves
1 AI moves fast. Docs, architecture records, and test coverage fall behind — silently — until someone notices months later.
2 CLAUDE.md says "update docs when you change code" — but without enforcement, it's guidance, not a rule. Claude will skip it under pressure.
3 Individual discipline doesn't scale. A PR review shouldn't be the last line of defense for documentation drift.
What we put in place
Runs on every PR to main — blocks merge if violated
agents/ modified → AGENTS.md must also be in the diff
runtime/ modified → SYSTEM_ARCHITECTURE.md must also be in the diff
workspace-config changed → .env.example must also be in the diff
AGENTS.md modified → PRESENTATIONS.md must also be in the diff
Lives in .github/workflows/doc-sync-check.yml — version controlled, runs in GitHub Actions, no trust required.
The CI check is what turns CLAUDE.md from a suggestion into a contract. It's also what enforces the Definition of Done at the merge level. Claude knows about it — and it changes behavior.
