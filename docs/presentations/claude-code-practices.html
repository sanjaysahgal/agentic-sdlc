Definition of Done

Code changed? These docs change too. Always.

What changedWhat must be updated
New agent built or modifiedAGENTS.md — persona, capabilities, inputs/outputs
New runtime behavior, routing, or data flowSYSTEM_ARCHITECTURE.md — architecture section
New backlog item completedBACKLOG.md — move to Completed
New solo-team shortcut takenDECISIONS.md — add entry with scale gap
New WorkspaceConfig field added.env.example — add with comment
Enforced rule: Do not ask if docs need updating — they always do if code changed. Do not defer. Update before marking done.
Without this table, doc drift is inevitable. Claude will happily write code and call it done without touching the docs. The table makes the expectation explicit and machine-enforceable.

.github/workflows/doc-sync-check.yml — runs on every PR to main. A PR that touches code without touching the relevant docs cannot merge.
The CI check is the enforcement layer for the Definition of Done table. Claude writes code correctly most of the time, but the CI catches the cases where it doesn't. You cannot ship a PR that adds an agent without documenting it. This is not optional.
Memory system

Corrections persist — same mistake never happens twice

feedback
When: user corrects an approach OR confirms a non-obvious approach worked
"Tests required for every behavior change"
— user had to point this out twice before it was saved
user
When: learning how the user thinks, what context they bring
"Learns best from concrete/observable first (process, command, file), then abstract"
project
When: learning about in-progress work, decisions, constraints, deadlines
"Redis persistence is a prerequisite before production deployment"
reference
When: learning where external information lives
"Brand tokens and design system live in agentic-brand repo"
Each type lives in its own file under ~/.claude/projects/<repo>/memory/, indexed by MEMORY.md. Loaded on every session. Corrections are saved immediately — not at session end.
The memory system is what makes Claude Code feel like a collaborator that learns, not a tool that resets. The key insight: corrections need to be saved immediately. If you wait until the end of the session, you forget. The CLAUDE.md instruction says: "After any correction, immediately save a feedback memory."
Subagent strategy

Main context for decisions. Subagents for everything else.

Delegate to subagents
  • →Broad codebase exploration — finding patterns, tracing data flow
  • →Research tasks — reading multiple files to answer a question
  • →Parallel analysis — two independent questions run simultaneously
  • →Large result sets — grep results that would flood main context
Keep in main context
  • →Implementation decisions — changes that require judgment
  • →Architecture discussions — reasoning over tradeoffs
  • →Anything where the answer informs the next action
  • →The actual writing of code or docs
Rule in CLAUDE.md: "Use subagents for exploration, research, and parallel analysis — keep the main context focused on implementation decisions. One task per subagent."
Context window is expensive. Flooding the main context with exploration results degrades the quality of implementation decisions. Subagents run in isolated contexts — their results come back as a clean summary, not as noise in the main thread.
Testing discipline

Tests are not optional and do not require prompting

The rule
Before building the second piece of any system, a test suite must exist for the first.

Every new agent behavior must have a corresponding test. A behavior with no test does not count as done.
Why it's in CLAUDE.md
Without explicit instruction, Claude will write code, call it done, and move on. The test requirement is not a reminder — it is part of the definition of completion.

Memory entries have structure: the rule, Why: (the reason — often a past incident), How to apply: (when it kicks in). Knowing why lets Claude judge edge cases instead of blindly following the rule.
The critical insight is "immediately" — corrections saved at end-of-session are often forgotten or incomplete. The CLAUDE.md instruction is explicit: "After any correction from the user, immediately save a feedback memory." Two memory categories feed this loop: feedback (corrections) and user (learned preferences and mental models).
What this produces

Consistent, auditable, self-correcting AI collaboration

01

Docs stay in sync with code — automatically

CLAUDE.md's Definition of Done + CI check means docs cannot drift. Every PR that touches code must touch the corresponding docs or it cannot merge.

02

Corrections compound across sessions

Every correction makes the next session better. The feedback loop means the collaboration improves continuously — not just within a session.

03

Constraints are explicit and enforced, not implied

CLAUDE.md makes every non-negotiable visible. No ambiguity about what "done" means, what's allowed, or what needs to be checked before starting.

The meta-point: these are not AI-specific practices. They are good engineering practices applied to AI collaboration. CLAUDE.md is a spec for how Claude should behave. The CI check enforces it. The memory system persists corrections. All the same patterns we use for human engineers.
