This is not about AI features. It's about workflow. How do you get consistent, high-quality output from Claude Code across a team, across sessions, over months? The answer is the same as for human engineers: structure, defined expectations, and feedback loops.
The thesis
Treat Claude like a senior engineer
joining the team.
Give it the same context you would give a person.
An onboarding doc with real constraints. A definition of done. Test requirements. A place to record what it learned.
The rest follows.
This is the whole deck in one slide. Everything that follows is just spelling out what "give it the same context" means in practice.
The real risk
AI moves fast. Without structure,
it optimizes for fast over durable.
01 Every session starts cold — same context re-explained, same mistakes repeated.
02 Code moves forward. Docs, tests, and architecture records fall behind.
03 Corrections made in a session vanish. The same guidance is needed next week.
These aren't hypotheticals. They're what happens by default. The practices in this deck each address one of these failure modes directly. They're not clever hacks — they're applied engineering discipline to AI tooling.
Practice 1

Give Claude an onboarding doc — CLAUDE.md

The challenge
Claude has no memory of your project when a session starts — no conventions, no constraints, no architecture context.
Without guidance, it will make reasonable-looking decisions that contradict decisions you made last sprint.
Every engineer on the team gets a different version of Claude depending on what they happen to say first.
What we put in place
// CLAUDE.md — checked into repo root, read every session
§1 What is this repo? — product identity + what it is NOT
§2 Core Principles — each rule + violation examples inline
§3 Before Writing Any Code — 4-question checklist, must pass
§4 Behavioral Rules — autonomous, durable, self-correcting
§5 Definition of Done — table: change type → required doc
§6 Subagent Strategy — delegate vs. keep in main
§7 Testing Discipline — rule + mocking pattern
Opening line: "Read this before doing anything. These are non-negotiable constraints."
Why it sticks Checked into the repo. Version-controlled. Reviewed in PRs. Not a personal habit — project infrastructure. Every engineer on the team gets the same Claude.
The opening line of CLAUDE.md is: "Read this before doing anything. These are non-negotiable constraints." That framing matters — it's not suggestions, it's the contract. The pre-code checklist is the highest-value section: it stops the most common category of drift before a line is written.
Practice 2

Make constraints specific — with violation examples

The challenge
"Be thoughtful" and "follow best practices" are not constraints. Claude interprets them generously in favour of the fastest path.
Vague rules produce inconsistent behavior — different engineers get different interpretations.
Without examples of what a violation looks like, the rule is abstract and ignored under time pressure.
What we put in place
// CLAUDE.md > Core Principles — each rule formatted as:
### Rule name
One sentence. Then violation examples inline:
**Violation examples to avoid:**
Single source of truth
vision.slice(0, 3000) // truncation
PRODUCT_SUMMARY.md // mirrors authoritative doc
Config over hardcoding
if (channel === "#eng-team") // string literal
config.mainChannel // from env/config file
Durable over fast
Flag the tradeoff explicitly before implementing. Never choose the brittle path silently.
Why it sticks Violation examples eliminate ambiguity. Claude cannot misinterpret "don't truncate documents" when it can see .slice(0, 3000) listed as a violation. The concrete example is the constraint.
The violation examples are the most important part. Abstract rules get rationalized away. A concrete anti-pattern that Claude can recognize in its own output is much harder to ignore.
Practice 3

Make "done" explicit — a table, not a feeling

The challenge
"Done" is implicit. Claude writes code, marks the task complete, and moves on — without touching the architecture doc, the decision record, or the env example.
Docs drift silently. Nobody notices until a new engineer reads stale architecture docs and builds on outdated assumptions.
Claude will not update docs unprompted. It needs to know it's part of the job.
What we put in place
CLAUDE.md › Definition of Done (Non-Negotiable)
What changedMust also update
New module or agent builtAGENTS.md — persona, inputs, outputs
New runtime behavior or data flowSYSTEM_ARCHITECTURE.md
New config field.env.example — with comment
Solo-team shortcut takenDECISIONS.md — what, why, scale gap
Backlog item completedBACKLOG.md — move to Completed
Do not ask if docs need updating — they always do if code changed. Do not defer to a later session.
Why it sticks The table is machine-enforceable. Each row maps directly to a CI check. The rule is not enforced by willpower — it's enforced at merge.
The key phrase in CLAUDE.md is: "Do not ask the human if docs need updating — they always do if code changed." That eliminates the decision entirely. The only question is which docs, and the table answers that.
Practice 4

Back the Definition of Done with a CI check

The challenge
Good intentions don't survive a deadline. Even with a clear DoD, Claude (and humans) skip doc updates under pressure.
There's no feedback signal at merge time. A PR with stale docs looks identical to one with current docs.
Enforcement that requires human review is not enforcement.
What we put in place
// .github/workflows/doc-sync-check.yml
on: pull_request: branches: [main]
jobs: doc-sync → runs-on: ubuntu-latest
agents/ modified → AGENTS.md must be in the diff — git diff --name-only origin/main
runtime/ modified → SYSTEM_ARCHITECTURE.md must be in the diff
workspace-config.ts modified → .env.example must be in the diff
❌ Error: agents/ was modified but AGENTS.md was not updated. PR blocked.
Why it sticks Can't merge without it. Zero reliance on memory or discipline. The check takes 10 seconds to write and pays off on every PR forever. This is the practice with the highest ROI.
The CI check is 40 lines of bash. It runs git diff against main and checks for file co-occurrence. Simple to write, impossible to accidentally bypass. Claude actually helps write and maintain it — which means it enforces its own compliance.
Practice 5

Make corrections survive the session boundary

The challenge
Claude corrected in session A has no memory of it in session B. The same mistake appears again next week.
The same explanations — how you like to work, what the project is, what was tried and failed — get repeated in every session.
Collaboration that doesn't accumulate learning isn't improving.
What we put in place
// ~/.claude/projects/<project-hash>/memory/
MEMORY.md ← index, loaded at every session start
feedback_tests.md ← correction: tests required before done
user_profile.md ← how this engineer learns + works
project_context.md ← current sprint state, blockers
// Each memory file — frontmatter + body:
---
name: Tests required for every behavior change
type: feedback
---
Never mark a task done without tests. Why: user had to
repeat this three sessions in a row before it was saved.
How to apply: check before every task completion.
Why it sticks Rule: save immediately on correction — not at session end. Loaded automatically at session start via MEMORY.md index. Corrections compound across every future session.
The most important detail: immediately. Corrections saved at session end are often incomplete or forgotten. The rule is: the moment Claude is corrected, it saves. The memory entry includes what, why, and how to apply — so Claude can judge edge cases rather than blindly following the rule.
Practice 6

Keep the main context lean — delegate exploration

The challenge
Exploration tasks — finding files, reading multiple modules, tracing data flow — flood the main context with raw results.
A context full of search results and intermediate reads degrades the quality of the implementation decision that follows.
Claude doing exploration inline means the main session is doing two jobs with one context window.
What we put in place
// CLAUDE.md › Subagent Strategy
"Use subagents for exploration, research, and parallel analysis.
Keep the main context focused on implementation decisions.
One task per subagent."
Delegate
Broad search — "find all callers of X across the repo"
Multi-file trace — "how does auth flow end to end?"
Parallel questions — fire both simultaneously
Keep in main
Writing code, making architectural calls
Anything that immediately informs the next action
Why it sticks Rule in CLAUDE.md: "Use subagents for exploration, research, and parallel analysis. Keep the main context for implementation decisions. One task per subagent." Subagents return a clean summary — not noise.
Context window is a finite resource and quality degrades as it fills. The explicit rule prevents Claude from defaulting to inline grep loops and read chains when a subagent would do the job more cleanly.
Practice 7

Tests are part of done — not a follow-up task

The challenge
By default, Claude writes code, calls it done, and moves on. Tests are optional unless explicitly required.
"We'll add tests later" is not a plan. At AI pace, the untested surface area compounds quickly.
Without a test gate, Claude's confidence in its own output is not calibrated to reality.
What we put in place
// CLAUDE.md › Testing Discipline (Non-Negotiable)
"Before building the second piece of any system,
a test suite must exist for the first.
A behavior with no test does not count as done."
Mocking pattern — also documented in CLAUDE.md
All external deps mocked — no real API calls in CI
vi.hoisted() for mocks used in vi.mock() factories
function() syntax (not arrow) for constructors used with new
// CI: runs on every push + PR to main
npx vitest run — zero real network calls, exits non-zero on fail
Why it sticks The rule is in CLAUDE.md — Claude checks it before marking any task done. The CI check runs tests on every PR. Neither humans nor Claude can merge untested code without the check failing.
The "second piece" framing is deliberate — it creates a concrete gate without requiring tests before a single function exists. But by the time the second module is being written, the first must be tested. This prevents the test debt from ever accumulating past one unit of work.
How it compounds

Each practice reinforces the others

CLAUDE.md Without: every session negotiates constraints from scratch → With: constraints are the starting point, not the conversation
Definition of Done Without: docs drift silently — noticed months later → With: doc updates are a named deliverable on every task
CI enforcement Without: DoD is aspirational — skipped under deadline pressure → With: DoD is structural — enforced at merge, no willpower needed
Memory system Without: corrections reset each session — team keeps re-explaining → With: corrections compound — AI collaboration improves over time
Subagent strategy Without: exploration bloats context — decision quality degrades → With: main context stays sharp — focused on the decision, not the search
Testing discipline Without: test debt compounds at AI pace — untested surface grows fast → With: every behavior is tested before the next one starts
None of these practices are expensive to put in place. CLAUDE.md takes an hour to write. The CI check takes 40 lines. The memory system is a folder and an index file. The total setup cost is a day. The payoff is months of compounding quality.
