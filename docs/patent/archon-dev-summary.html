Archon Dev

System and Method for Orchestrating Stateless AI Agents Across Software Development Lifecycle Phases Using Version Control as Authoritative State

The Problem

Software development lifecycles require significant human coordination across phases: product specification, design, engineering, implementation, and deployment. No existing tool orchestrates AI agents coherently across all phases. The market is fragmented:

GitHub Copilot WorkspaceSingle-repo code generation. No phase awareness, no spec validation, no multi-team support.
LangChain / AutoGenInfrastructure primitives only. Stateful processes, no SDLC understanding, no multi-tenant model.
Linear / JiraProprietary databases outside version control. No AI orchestration. Manual per-team setup.
All existing toolsNo system combines version-controlled state, stateless agents, synchronous conflict detection, and zero-code multi-tenancy.

The Invention

Archon Dev is a config-driven, multi-tenant platform that orchestrates stateless AI agents across every phase of the software development lifecycle. Version control is the sole state store — no separate database. A team onboards by providing a single configuration file. No code changes required.

The platform operates across three layers: a human interface (Slack, web, or any messaging system), a stateless agent runtime (AI agents as pure functions), and a state layer (GitHub as the authoritative record of all phases, specs, and transitions). These layers are explicitly decoupled — swapping the interface requires no changes to agent or routing logic.

The key insight: Git is already a state machine. Branch existence, file presence, and merge status are sufficient to encode the entire lifecycle of a software feature — no workflow engine or database needed. Every phase transition is a standard Git operation, visible and auditable by any developer using standard tooling.

What Is Claimed — 12 Embodiments

1 — Version Control as SDLC State Machine

Lifecycle phase is inferred entirely from Git state (branch existence, file merge status) — no external database. Phase transitions are standard Git operations: branch creation, pull request merge.

2 — Stateless AI Agent Architecture with Fresh Context Injection

Every agent is a stateless function. Each invocation receives a programmatically-built system prompt, an externally-persisted conversation history, and context filtered from live repository documents by a secondary language model (relevance filter). No agent state persists between calls — given identical inputs, an agent produces identical outputs.

3 — Synchronous Conflict Detection with Re-Read Verification

Every specification draft is validated against upstream documents before saving. Conflicts block the save. When the human claims to have resolved a conflict, the agent re-reads the upstream document from the repository to verify the update was actually committed — it does not take the human's word for it.

4 — Phase-Aware Agent Behavior Switching

A single agent serves both draft mode (collaborative spec shaping) and approved mode (frozen baseline, re-approval required for changes) by switching system prompt content based on detected phase. No separate agent instances are deployed per mode.

5 — Config-Driven Multi-Tenant Architecture

All product-specific coordinates (product name, repo, channel names, spec paths, routing rules) live in a WorkspaceConfig object loaded from environment variables. No product-specific literals appear anywhere in platform source code. A new team onboards by providing a new .env file — no code changes. CI enforcement prevents regression.

6 — Decoupled CI/CD Platform with Configuration-Only Onboarding

All pipeline logic (build, test, deploy, post-deploy validation) lives in a central platform repository. Application repositories call it via a single workflow file. No pipeline logic is duplicated. Improvements to the platform are automatically inherited by all onboarded applications.

7 — Structured Open Question Tagging for Automated Phase Gating

Open questions in specs are tagged [type: product|design|engineering] [blocking: yes|no]. The type enables automated routing to the right reviewer. The blocking flag gates phase advancement — a spec with unresolved blocking questions cannot be approved.

8 — Decoupled Interface-Agnostic Human Interaction Layer

The interface layer (Slack, web app, Teams, etc.) is responsible only for receiving input and rendering output — no business logic, no SDLC knowledge. Swapping one interface for another requires zero changes to agent, routing, or state management components.

9 — Version-Control-Derived Monitoring Dashboard

A real-time dashboard derives all state directly from the version control repository — no separate data store, no synchronization. Provides portfolio view, blocking issues, conflict log, agent activity feed, and cross-workspace SDLC health. Any historical view is reconstructable from Git history.

10 — Authoritative Specification Chain with Full Upstream Context

Every downstream agent must load the complete set of upstream approved specs before producing output. Design agents read the product spec. Architect agents read both. Engineer agents read all three. Conflict detection enforces chain integrity end-to-end — no downstream artifact can contradict any upstream artifact.

11 — Cross-Phase Escalation Protocol

When a downstream agent detects a blocking question owned by an upstream phase, it invokes the upstream agent directly in the current conversation thread — with the question, relevant spec context, and full conversation history as a primer. The upstream agent responds with a concrete proposal. No manual relay, no context loss.

12 — Proactive Phase Orchestration with Handoff Gating

A dedicated orchestration agent monitors Git state across all active features, proactively notifies stakeholders when phases are ready to advance, blocks handoffs when blocking questions remain unresolved, detects stalls, and owns the canonical routing table as the sole source of truth for phase-to-agent mappings.

Archon Dev is not a concept — it is a running system. The following embodiments are implemented and operational as of the filing date:

  • Embodiment 6: Decoupled CI/CD platform is operational, with Health360 as the first customer application onboarded via configuration only.
  • Embodiment 7: Structured open question tagging with blocking gates is enforced in the pm agent and UX Design agent.

The platform currently serves Health360 as customer zero — a real consumer health application being built end-to-end through Archon Dev. The full autonomous pipeline (spec → design → engineering → code → QA → production) is the active development target.
