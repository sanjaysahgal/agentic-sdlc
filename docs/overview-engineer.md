# agentic-sdlc — Engineering Technical Reference

## How the system works

agentic-sdlc is a spec-driven, agent-assisted SDLC pipeline. The full flow:

```
PM → product spec → design spec → engineering spec → work items → code → QA → shipped
```

Each transition is a human approval gate. Each phase is supported by an AI agent that reads prior specs from GitHub and assists the human through conversation. Agents are Claude-backed (claude-sonnet-4-6 for reasoning, claude-haiku-4-5 for classification).

## What you receive as an engineer

By the time work reaches engineering, you have:
- An approved `<feature>.product.md` — what the feature does, acceptance criteria, edge cases
- An approved `<feature>.design.md` — screen inventory, user flows, component list
- An approved `<feature>.engineering.md` — data model changes, API contracts, build sequence, technical decisions
- GitHub Issues generated by the pgm agent — one per discrete unit of work, linked to the engineering spec section

You build against the spec. If something in the spec is wrong, you raise it — you don't make unilateral product or design decisions in code.

## Repository structure (agentic-sdlc)

```
agents/                ← AI agent definitions (system prompts, intent detection)
  concierge.ts         ← entry point agent (main channel)
  pm.ts                ← product spec agent
  concierge.ts         ← front desk agent
runtime/
  claude-client.ts     ← Anthropic SDK wrapper (runAgent)
  github-client.ts     ← Octokit wrapper (readFile, saveDraftSpec, createSpecPR, getInProgressFeatures)
  context-loader.ts    ← loads product vision, feature conventions, arch, current draft per message
  conversation-store.ts ← in-memory + disk persistence for thread history and confirmed agents
  agent-router.ts      ← Claude Haiku intent classifier, phase detector, agent type definitions
interfaces/
  slack/
    app.ts             ← Slack Bolt app, event routing
    server.ts          ← entry point
    handlers/
      message.ts       ← feature channel message handler
      general.ts       ← main channel (concierge) handler
```

## GitHub as state machine

Phase state is derived entirely from GitHub — no separate database:

| Condition | Phase |
|---|---|
| Branch `spec/<feature>-product` exists, file not on `main` | Product spec in progress |
| `specs/features/<f>/<f>.product.md` on `main` | Product spec approved |
| Branch `spec/<feature>-design` exists, file not on `main` | Design in progress |
| `specs/features/<f>/<f>.design.md` on `main` | Design approved |
| `specs/features/<f>/<f>.engineering.md` on `main` | Engineering spec approved |

## Running locally

```bash
cd agentic-sdlc
cp .env.example .env   # fill in all 5 values
npm install
npm run dev
```

Required env vars: `SLACK_BOT_TOKEN`, `SLACK_SIGNING_SECRET`, `SLACK_APP_TOKEN`, `ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, `GITHUB_OWNER`, `GITHUB_REPO`

The bot uses Slack Socket Mode — no public URL or ngrok needed.

## Key design decisions

**Context loaded fresh per message.** `context-loader.ts` calls GitHub on every message — product vision, feature conventions, system architecture, current draft. No caching. This ensures agents always see the latest spec state, at the cost of ~200ms GitHub API latency per message.

**Draft = source of truth, not conversation history.** The pm agent reads the saved draft from the feature branch at the start of every message. Conversation history is supplementary context. A bot restart loses conversation history but not the draft — the agent continues correctly.

**Confirmed agent persisted to disk.** `.confirmed-agents.json` maps `threadTs → agentType`. Written on every confirmation, loaded on startup. Prevents re-asking for agent confirmation after bot restarts. Production target: Redis.

**Two Claude models.** `claude-sonnet-4-6` for all agent reasoning (pm, architect, etc.). `claude-haiku-4-5` for intent classification only — faster and cheaper for a task that doesn't need deep reasoning.

**Framework-agnostic runtime.** `agents/` and `runtime/` have no Slack dependency. `interfaces/slack/` is one delivery mechanism. Adding a GitHub Actions trigger or web portal means adding a new `interfaces/` subdirectory without touching agent logic.

## Production gaps (current)

| Gap | Current | Production target |
|---|---|---|
| Conversation history | In-memory (lost on restart) | Redis or Neon |
| Confirmed agent | Local JSON file | Redis |
| Agent triggers | Manual (Slack message) | GitHub webhook → GitHub Action |
| Phase detection | Branch names + file existence | Same (no change needed) |
| Spec review | Manual GitHub PR review | spec-validator agent runs on PR open |
